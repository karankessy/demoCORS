🌐 CORS Policy Demo

Three servers running simultaneously — see CORS in action

🌐 :3000 — Frontend (this page) 🚫 :4000 — Backend (no CORS) ✅ :4001 — Backend (with CORS)
How it works:
This page is served from http://localhost:3000. Two separate backend servers are running at the same time:
• localhost:4000 — has no CORS headers (browser will block responses)
• localhost:4001 — has CORS enabled for localhost:3000 (browser will allow responses)

Different ports = different origins. The browser enforces CORS on every cross-origin fetch().
① Simple Requests (GET)

🔵 Same-Origin Always Works

GET → localhost:3000 (same origin, no CORS involved)

Request stays on localhost:3000. No CORS check — browser allows it by default.

🔴 Cross-Origin → :4000 Blocked

GET → localhost:4000 (no CORS headers on this server)

Cross-origin request to :4000. Server has NO CORS headers → browser blocks JavaScript from reading the response.

🟢 Cross-Origin → :4001 Allowed

GET → localhost:4001 (CORS enabled for :3000)

Cross-origin request to :4001. Server sends Access-Control-Allow-Origin header → browser allows JavaScript to read the response.

② Preflight Requests (POST with application/json)

🔴 Preflight → :4000 Blocked

POST with Content-Type: application/json triggers OPTIONS preflight

POST with JSON body is NOT a "simple request". Browser sends OPTIONS preflight first. Without CORS headers → preflight fails, actual POST is never sent.
Show Response Headers

🟢 Preflight → :4001 Allowed

Server responds to OPTIONS with correct CORS headers

POST with JSON body triggers OPTIONS preflight. Server at :4001 responds with CORS headers → preflight passes → actual POST succeeds.
